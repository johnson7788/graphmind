整体流程
1. 文本切块                       │
      │    文章A → 切成若干 text_units    │
      │    文章B → 切成若干 text_units    │
      │    所有块混在一起                  │
      └──────────────┬────────────────────┘
                     ↓
      ┌───────────────────────────────────┐
      │ 2. 实体关系抽取（extract_graph）   │
      │    对所有 text_units 逐一调 LLM   │
      │    文章A的 ←→ 文章B的实体         │
      │    如果在同一个块/相关上下文出现    │
      │    就可能产生跨文章的关系           │
      └──────────────┬────────────────────┘
                     ↓
      ┌───────────────────────────────────┐
      │ 3. 摘要合并（summarize_descriptions）│
      │    同名实体去重                    │
      │    "雍正"在A里出现3次 B里出现2次   │
      │    → 合并成一个实体 + 综合描述     │
      └──────────────┬────────────────────┘
                     ↓
      ┌───────────────────────────────────┐
      │ 4. 社区聚类（cluster_graph）       │
      │    用整张图（已合并的实体+关系）   │
      │    跑 Leiden 算法                  │
      │    可能：文章A的「八旗制度」和      │
      │    文章B的「兵制改革」聚到同社区    │
      └──────────────┬────────────────────┘
                     ↓
      ┌───────────────────────────────────┐
      │ 5. 社区报告 + 向量化               │
      └─────────────────────────

1. settings.yaml — entity_types: [组织,人物,地理位置,事件]
2.关系没有预定义类型，和实体不一样。流程是：


    1. LLM 先找出所有实体（按 entity_types 约束）
    2. LLM 扫描所有实体对，判断哪些"明确相关"
    3. 对每对相关实体，输出：
       - source / target（两个实体名）
       - description（自然语言描述关系，如"蔡崇信担任阿里巴巴集团董事会主席"）
       - strength（1-10 数值分）

3.检索

    | 你想问的问题              | 选哪个   |
    |---------------------------|----------|
    | "XX 和 YY 之间发生了什么" | 本地搜索 |
    | "这篇文章整体在讲什么"    | 全局搜索 |
    | "文档里有没有提到 ZZ"     | 基础 RAG |

    基础 RAG（Basic RAG）— 不用图谱
    原理：把问题向量化 → 在 text_units 向量库里找最相似的文本块 → 直接喂给 LLM
    适应场景：关键词匹配型问题，图谱可能漏掉的细节

    全局搜索（Global Search）— 宏观理解
    原理：按 community 分组 → 把 community_reports（索引阶段 LLM 已经写好的社区摘要）按相关性拼起来 → 让 LLM 综合多个报告回答
    适应场景：概括性问题，需要宏观视角

    本地搜索（Local Search）— 默认推荐
    原理：提取问题中的实体名 → 在图谱中定位 → 展开邻居 → 拿相关 text_units 做向量匹配 → 图谱上下文 + 文本块一起喂给 LLM
    适应场景：具体问题，需要精确事实
    问题 → 向量匹配 10 个实体 → 顺藤摸瓜找社区报告（3000 tokens） + 关系（3000 tokens） + 原文（6000 tokens）
        → 按比例拼进 12000 token 预算 → 一起喂给 LLM

本地检索中，关系如何获取
两轮收集，按 rank 排序，top_k 截断
    选中实体: [蔡崇信, 阿里巴巴, 吴泳铭]

    ──── 第一优先级：内部关系 ────
    在选中实体之间找关系：
      蔡崇信 ←→ 阿里巴巴  (关系 rank=9)
      蔡崇信 ←→ 吴泳铭    (关系 rank=5)
      吴泳铭 ←→ 阿里巴巴  (关系 rank=7)
      → 按 rank 降序排列

    ──── 第二优先级：外部关系 ────
    从选中实体出去的边：
      蔡崇信 ←→ 杭州      (rank=6)
      吴泳铭 ←→ 淘天集团  (rank=8)
      吴泳铭 ←→ 云智能集团 (rank=7)
      阿里巴巴 ←→ 杭州     (rank=4)

      外部实体「杭州」被 2 个选中实体关联 → mutual 优先，排前面
      → 按 mutual 数降序，同层内按 rank 降序

    ──── 截断 ────
    top_k_relationships = 10
    只取前 10 条

一句话总结

    | 搜索类型 | 图谱 | 向量 | 核心机制              |
    |----------|------|------|-----------------------|
    | Local    | ✓    | ✓    | 图谱遍历 + 向量相似度 |
    | Global   | ✓    | ✗    | 社区报告聚合          |
    | Basic    | ✗    | ✓    | 纯向量 RAG            |

社区报告是 GraphRAG 索引 pipeline 的最后一步（create_community_reports），用 LLM 自动化批量生成的。分三层来讲：



    第一层：什么是"社区"

    Leiden 算法把知识图谱按连接密度自动划分成若干子图，每个子图就是一个"社区"：


    整个图谱
      ├── 社区 0（n 个实体紧密关联）
      │     实体A ←→ 实体B
      │     实体B ←→ 实体C
      ├── 社区 1
      │     实体D ←→ 实体E
      └── 社区 2 ...


    越往上层（community_level 越高），社区越大越宏观。



    第二层：怎么生成的


    create_community_reports 工作流：

    对每个社区:
      → 收集该社区的: 实体列表 + 关系列表 + 文本片段
      → 拼成一段 prompt，调用 LLM
      → LLM 输出结构化 JSON:
         {
           "title": "社区标题",
           "summary": "一段概述",
           "rating": 7.5,           ← 重要性评分
           "rating_explanation": "...",
           "findings": [
             {"summary": "...", "explanation": "..."},
             ...
           ]
         }
      → 存入 community_reports.parquet


    你在「数据浏览」tab 里看到的每一条就是一次 LLM 调用的结果。



    第三层：Global Search 怎么用这些报告

    全局搜索的逻辑是：


    用户问题 "这篇文章的核心主题是什么？"
      → 把问题向量化
      → 在 community_reports 的 full_content 里做向量匹配
      → 选出最相关的 N 个社区报告
      → 拼在一起喂给 LLM：综合回答


    所以社区报告本质上是一组预生成的知识摘要，让全局搜索不需要现场遍历所有实体，直接读摘要就能回答宏观问题。



    两个 prompt 的区别

    community_report_graph.txt 和 community_report_text.txt 对应社区报告的不同来源：

    | prompt   | 输入来源              | 侧重               |
    |----------|-----------------------|--------------------|
    | graph 版 | 实体 + 关系（图结构） | 实体间关联、影响力 |
    | text 版  | 原始文本片段          | 具体内容、时间线   |

    LLM 拿到后按统一格式输出 JSON，两部分合在一起存成一个社区报告。



    你日志里看到的 1/14 ... 14/14 就是在对 14 个社区逐一生成了 14 份报告，每份报告里又有若干条 findings。
